# Parallelization Plan — TaskFlow API (EXAMPLE)

> Generated by `/bmad-planning-orchestrator:bmad-parallel-plan`
> Intent: **Create** (first wave plan)
> Date: 2026-06-19  •  Max parallel: **3**  •  Waves: **6**

This is a **concrete example** of the artifact `bmad-parallel-plan` emits for the
TaskFlow API walkthrough documented in `docs/examples/index.md`. It is illustrative
only — the skill generates the real plan automatically from your story files.

This plan turns the ready-for-dev backlog into conflict-free concurrent **waves**. Each
story gets an isolated git worktree branch with a disjoint file scope. Waves run in
order; inside a wave, stories run concurrently. **This is a plan — your external dev
tools create the worktrees, run the agents, merge, review, and open the PR. This plugin
does none of that.**

## Waves

### Wave 1 — base branch `main`

Run these stories **concurrently**, each in its own isolated worktree:

| Story | Title | Worktree branch | Disjoint file scope |
|-------|-------|-----------------|---------------------|
| 1.1 | Project Scaffolding and CI | `story/1.1-project-scaffold` | `package.json`, `tsconfig.json`, `.github/workflows/ci.yml`, `src/` (skeleton only) |
| 1.2 | Base DB Migration | `story/1.2-base-migration` | `src/db/migrations/001_create_schema.sql`, `src/db/data-source.ts` |

Why these can co-run: file scopes are disjoint; neither touches a shared/cross-cutting
module; neither has any `depends_on`. Note that 1.1 owns `src/` broadly — in practice,
1.1 writes only the top-level skeleton files (index.ts, app.ts) and the 1.2 developer
adds files inside `src/db/`. The architecture document and story scopes both reflect this
split.

**Merge sequence (ascending story id):**

1. Merge `story/1.1-project-scaffold` -> `integration/wave-1`
2. Merge `story/1.2-base-migration` -> `integration/wave-1`
3. **Integration review checkpoint** on `integration/wave-1` — verify scaffold and base
   migration work together (run `npm run build` and `npm run db:migrate`).
4. Open PR: `integration/wave-1` -> `main`.
5. Wave 2 branches off the merged `main`.

### Wave 2 — base branch `main` (after wave 1 merges)

| Story | Title | Worktree branch | Disjoint file scope |
|-------|-------|-----------------|---------------------|
| 2.1 | User Model and Migration | `story/2.1-user-model` | `src/db/migrations/002_create_users.sql`, `src/db/models/user.entity.ts`, `src/types/auth.types.ts` |

Why only one story: 2.1 touches `src/types/` (a shared cross-cutting module). No other
ready story can safely co-run with it. 3.1.project-model also depends on 2.1 (foreign
key reference), so it cannot enter until 2.1 is done.

**Merge sequence:**

1. Merge `story/2.1-user-model` -> `integration/wave-2` (single story; merge is trivial)
2. **Integration review checkpoint** — verify user entity and migration are consistent with
   base schema from wave 1.
3. Open PR: `integration/wave-2` -> `main`.
4. Wave 3 branches off the merged `main`.

### Wave 3 — base branch `main` (after wave 2 merges)

| Story | Title | Worktree branch | Disjoint file scope |
|-------|-------|-----------------|---------------------|
| 2.2 | Auth Service | `story/2.2-auth-service` | `src/services/auth-service.ts`, `src/services/__tests__/auth-service.test.ts` |
| 3.1 | Project Model and Migration | `story/3.1-project-model` | `src/db/migrations/003_create_projects.sql`, `src/db/models/project.entity.ts` |
| 4.1 | Task Model and Migration | `story/4.1-task-model` | `src/db/migrations/004_create_tasks.sql`, `src/db/models/task.entity.ts` |

Why these can co-run:
- 2.2 depends on 2.1 (user model), which is done. Its scope is in `src/services/` only.
- 3.1 depends on 2.1 (user foreign key). Its scope: one migration slot + one entity file.
- 4.1 depends on 3.1 — wait, does it? The dependency map for 4.1 lists `depends_on: [3.1]`.
  However, in the topological sort, 3.1 and 4.1 both become ready in the same generation
  because 4.1's dependency on 3.1 has no ordering constraint that places 3.1 in an *earlier*
  wave — unless the story explicitly says so.

  **Planner note:** For this example, the story-author declared `4.1 depends_on: [3.1]` to
  enforce that the tasks table's project FK resolves after the projects table exists. The wave
  algorithm therefore places 4.1 one wave *after* 3.1. Because 3.1 is in wave 3 and 4.1
  depends on it, 4.1 should be wave 4. See correction in wave 4 below.

  After re-sort: wave 3 = {2.2, 3.1, 5.1} because 5.1 (comment model) is also ready at
  this generation and is file-disjoint.

**Revised wave 3:**

| Story | Title | Worktree branch | Disjoint file scope |
|-------|-------|-----------------|---------------------|
| 2.2 | Auth Service | `story/2.2-auth-service` | `src/services/auth-service.ts`, `src/services/__tests__/auth-service.test.ts` |
| 3.1 | Project Model | `story/3.1-project-model` | `src/db/migrations/003_create_projects.sql`, `src/db/models/project.entity.ts` |
| 5.1 | Comment Model | `story/5.1-comment-model` | `src/db/migrations/005_create_comments.sql`, `src/db/models/comment.entity.ts` |

All three touch different migration slots. None touch `src/types/` or `src/middleware/`.
None share file scope. Dependency-safe: all `depends_on` resolved in waves 1–2.

**Merge sequence:**

1. Merge `story/2.2-auth-service` -> `integration/wave-3`
2. Merge `story/3.1-project-model` -> `integration/wave-3`
3. Merge `story/5.1-comment-model` -> `integration/wave-3`
4. **Integration review checkpoint** — run full migration sequence (001 through 005);
   verify auth-service tests pass against the user entity from wave 2.
5. Open PR: `integration/wave-3` -> `main`.
6. Wave 4 branches off the merged `main`.

## Merge order rationale

Within a wave every branch is file-disjoint, so the branches can merge in any order
without textual conflict. Ascending story id is imposed for **predictable, reproducible
review**: lower ids are typically more foundational, so the integration branch grows
outward from the most foundational change.

One reviewed PR per wave (`integration/wave-N -> main`) keeps `main` green instead of
racing N independent PRs. Each wave starts from a known-good `main`, so later waves
automatically inherit earlier waves' changes and their prerequisites hold.

## Handoff

For each wave, hand your dev tool: the worktree branch names, each story's file scope,
and the merge sequence above. The story files in `bmad-output/stories/` are the
authoritative context objects. Their **Acceptance Criteria, Dev Notes, and Testing
sections are LOCKED** — the dev tool implements against them but must not edit them.

The `handoff-manifest.json` produced by `bmad-handoff` carries all of this in
machine-readable form (wave, ownedScope, dependencies, AC summary, lockedSectionsNote)
for any downstream runner that prefers structured input over Markdown.
